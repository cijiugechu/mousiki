# Rust/C Opus Benchmark Method

This document records the current benchmark method used to compare the Rust
implementation in this repository against the upstream C implementation under
`opus-c/`.

The goal is to measure codec runtime, not CLI startup or file I/O overhead.

## Scope

The current harness compares:

- `encode`: Rust vs C encoding the same in-memory PCM frames
- `decode`: Rust vs C decoding the same packet corpus

It does not use `hyperfine` or any external process-level timing tool as the
primary measurement mechanism. Timing happens inside the benchmark binaries so
the numbers reflect codec work more directly.

`hyperfine` is used as a secondary cross-check to validate process-level
wall-clock differences between Rust and C decode binaries.

## Components

Rust side:

- `src/bin/codec_bench.rs`

C side:

- `ctests/codec_bench.c`
- `ctests/CMakeLists.txt`

Wrapper script:

- `scripts/bench_codec_compare.sh`

## Design

The comparison is split into two paths.

### Encode

- Both implementations read the same raw PCM input.
- PCM is preloaded into memory before timing begins.
- Each timed iteration creates a fresh encoder and encodes the same frame set.
- Timing excludes input file reading and benchmark argument parsing.

### Decode

- Decode must consume the same packet stream on both sides.
- To avoid Rust packet shape influencing the result, the shared decode corpus is
  generated by the C benchmark first.
- The packet corpus format is a small benchmark-only container with magic
  `OPUSBEN1`.
- Each timed iteration creates a fresh decoder and decodes the same packet list.

This asymmetry is deliberate:

- `encode` compares how fast each side produces packets from the same PCM
- `decode` compares how fast each side consumes identical packets

## Packet Corpus Format

The decode corpus written by `packets` mode is:

- 8-byte magic: `OPUSBEN1`
- `u32` sample rate, little-endian
- `u16` channels, little-endian
- `u16` frame size, little-endian
- `u32` application, little-endian
- `i32` bitrate, little-endian
- `u8` complexity
- `u8` bitrate mode
- `u16` reserved
- repeated packet records:
  - `u16` packet length
  - packet bytes

This format is benchmark-only. It is not intended as a stable public file
format.

## Output Metrics

Both binaries print either text or CSV.

The main fields used for comparison are:

- `median_ns_per_frame`
- `p95_ns_per_frame`
- `median_packets_per_sec`
- `median_realtime_x`

For cross-implementation comparison, `median_ns_per_frame` is the primary
metric. Lower is better.

## Current Matrix

The current comparison matrix uses four representative cases:

1. `voip_16k_mono_20ms_24kbps`
2. `audio_48k_mono_20ms_64kbps`
3. `audio_48k_stereo_20ms_128kbps`
4. `audio_48k_stereo_60ms_128kbps`

The stereo source PCM currently used is:

- `testdata/ehren-paper_lights-96.pcm`

Derived inputs are generated with `ffmpeg`:

- 48 kHz mono
- 16 kHz mono

PCM tails that do not form a complete frame are ignored. The harness only times
complete frames.

## Build Configuration

Rust:

- `cargo build --release --bin codec_bench`

C:

- `cmake -S ctests -B ctests/build-bench -DOPUS_CTESTS_FIXED_POINT=OFF -DOPUS_CTESTS_ENABLE_FLOAT_API=ON`
- `cmake --build ctests/build-bench --target opus_codec_bench`

Important:

- The harness builds and links against `opus-c`, but it does not modify files
  under `opus-c`.
- The current comparison is for the floating-point build, not `fixed_point`.

## Quick Run

For a single-case compare, use:

```bash
./scripts/bench_codec_compare.sh \
  --input ./testdata/ehren-paper_lights-96.pcm \
  --sample-rate 48000 \
  --channels 2 \
  --frame-size 960 \
  --application audio \
  --bitrate 64000 \
  --complexity 10 \
  --bitrate-mode cvbr \
  --warmup 3 \
  --measure 10
```

This script:

1. builds both benchmark binaries
2. generates a C packet corpus
3. runs Rust/C encode on the same PCM
4. runs Rust/C decode on the same corpus
5. prints one CSV table

## Hyperfine Validation (Decode)

Use this when you want an independent process-level A/B check for Rust vs C
decode speed on exactly the same packet corpus.

1. Build the benchmark binaries:

```bash
cargo build --release --bin codec_bench
cmake -S ctests -B ctests/build-bench -DOPUS_CTESTS_FIXED_POINT=OFF -DOPUS_CTESTS_ENABLE_FLOAT_API=ON
cmake --build ctests/build-bench --target opus_codec_bench
```

2. Generate a shared decode corpus from the C benchmark (example case:
   `audio_48k_stereo_20ms_128kbps`):

```bash
ctests/build-bench/opus_codec_bench packets \
  --input testdata/ehren-paper_lights-96.pcm \
  --output target/codec-bench/decode-corpus-48000-2ch-960-audio-128000.opusbench \
  --sample-rate 48000 \
  --channels 2 \
  --frame-size 960 \
  --application audio \
  --bitrate 128000 \
  --complexity 10 \
  --bitrate-mode cvbr \
  --max-frames 4000
```

3. Run `hyperfine` against Rust/C decode commands:

```bash
hyperfine --warmup 3 --runs 20 \
  --export-json target/codec-bench/profile-tuning/hyperfine_rust_vs_c_decode.json \
  --export-markdown target/codec-bench/profile-tuning/hyperfine_rust_vs_c_decode.md \
  'target/release/codec_bench decode --packets target/codec-bench/decode-corpus-48000-2ch-960-audio-128000.opusbench --warmup 0 --measure 10 --max-frames 4000 --format csv --no-header > /dev/null' \
  'ctests/build-bench/opus_codec_bench decode --packets target/codec-bench/decode-corpus-48000-2ch-960-audio-128000.opusbench --warmup 0 --measure 10 --max-frames 4000 --format csv --no-header > /dev/null'
```

Notes:

- Keep both commands on the same packet corpus path.
- Keep decode arguments (`--measure`, `--max-frames`) identical.
- Redirect output to `/dev/null` to avoid terminal output overhead.
- Prefer at least 20 runs and inspect the exported markdown/json for outliers.

## Latest Snapshot

From the current run recorded in `target/codec-bench/results/compare.csv`:

- Encode is close to C overall:
  - Rust is faster by about `10.1%` on `voip_16k_mono_20ms_24kbps`
  - Rust is faster by about `2.9%` on `audio_48k_mono_20ms_64kbps`
  - Rust is slower by about `12.8%` on `audio_48k_stereo_20ms_128kbps`
  - Rust is slower by about `9.1%` on `audio_48k_stereo_60ms_128kbps`

- Decode is consistently slower than C:
  - about `57.1%` slower on `voip_16k_mono_20ms_24kbps`
  - about `64.2%` slower on `audio_48k_mono_20ms_64kbps`
  - about `63.1%` slower on `audio_48k_stereo_20ms_128kbps`
  - about `64.4%` slower on `audio_48k_stereo_60ms_128kbps`

That pattern suggests decode is currently the higher-value profiling target.

## Limitations

This method is good for relative implementation comparison, but it is not yet a
publication-grade performance methodology.

Current limitations:

- no CPU affinity pinning
- no frequency governor control
- no hardware-counter profiling
- one machine, one build configuration
- no fixed-point comparison yet
- no separation of top-level Opus overhead vs CELT/SILK internal hot paths

## Recommended Next Step

If the goal is optimization rather than just reporting, use this harness to
focus on decode first, then add targeted profiling for:

- top-level `opus_decode`
- CELT decode hot paths
- shared float conversion / buffer movement paths
